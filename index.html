Areas de GIT

Working Directory

El directorio de trabajo hace referencia a la carpeta donde creamos el proyecto, muy rara vez utilizaremos la linea de comandos y trabajaremos principalmente con nuestro editor predilecto.

Staging Area (INDEX)

El area de control de cambios corresponde a un Archivo de la carpeta .git que identifica cuales son aquellos elementos del proyecto a los que estamos haciendole seguimiento. Es el area donde se realiza la captura del codigo (SNAPSHOT) para ser confirmada mas adelante.

Repository (LOCAL/REMOTE)

Es el area donde se almacenan los cambios confirmados mediante commits en formato BLOB (Binary Large Object). Basicamente es lo nos permite acceder a puntos anteriores dentro del proyecto, ya que podemos apuntar a los diferentes nodos dentro del historial de confirmaciones.

Apuntadores GIT

HEAD

Apuntador movil que indica donde nos encontramos en el historial de cambios y ademas permite desplazarnos entre los diferentes commits. Tambien funciona como referente a la hora de realizar comparaciones, como por ejemplo HEAD^^ y HEAD~2 indican que hacemos referencia a 2 commits anteriores o ancestros.

BRANCH

Son apuntadores dinamicos que se ubican en el ultimo commit del historial de cambios, podemos crear todas las que necesitemos segun las variantes que tenga nuestro proyecto. Sirven como referentes ya que los nodos de git desconocen cuales son los commits posteriores. Se utilizan principalmente para probar caracteristicas nuevas, realizar propuestas o cambios que no sabemos si funcionaran en el proyecto final.

TAG

Son apuntadores estaticos que se utilizan principalmente para el versionado del proyecto, ya que hacen referencia al commit donde se generaron. Facilita la navegacion dentro del historial ya que funcionan como puntos de control nombrados, Generalmente se identifican de la siguiente manera v0.0.0 donde el primer numero indica la version principal, el segundo la compilacion o caracteristica agregada y el ultimo alguna correccion o revision del codigo.

STASH

Apuntador de la zona temporal de cambios, se utiliza principalmente cuando tenemos cambios pendientes que no ameritan la realizacion de un commit y necesitamos cambiar de rama. Genera una referencia al commit actual y el INDEX del mismo; de esta manera cuando se reaplican los cambios almacenados, se realiza una operacion merge a la inversa. En caso de trabajar con archivos que a los que no estemos haciendole seguimiento es necesario realizar el git add.
